# Repository Guidelines

## Project Structure & Module Organization
This repository is a Go module (`linuxvm`) for `revm`, with entrypoints under [`cmd/`](cmd/) and shared packages under [`pkg/`](pkg/).

- `cmd/revm`: main CLI binary.
- `cmd/revm-helper`: Linux wrapper used in packaged builds.
- `cmd/krun-runner`, `cmd/guest-agent`: VM helper binaries.
- `pkg/*`: core subsystems (networking, filesystem, VM builder, service APIs).
- `docs/`: user-facing mode and API docs.
- `pkg/static_resources/`: packaged rootfs/raw disk assets.
- `out/`: transient build output generated by `build.sh`.

## Build, Test, and Development Commands
Use Go 1.25.5 (matches CI).

- `bash ./build.sh`: full release-style build; fetches dependencies/assets and creates `revm-<OS>-<ARCH>.tar.zst`.
- `bash ./build.sh --dirty`: reuse cached `/tmp/.deps` to speed up iterative builds.
- `go build ./cmd/revm`: quick local compile of the main CLI.
- `go test ./...`: run unit tests across all packages.
- `golangci-lint run`: run linters and formatters defined in `.golangci.yml`.

## Coding Style & Naming Conventions
- Follow standard Go formatting: `gofmt` and `goimports` (enforced by `golangci-lint`).
- Keep package names short, lowercase, and domain-focused (for example `vmbuilder`, `filesystem`).
- Use tabs (Go default indentation) and idiomatic Go naming (`CamelCase` exported, `camelCase` internal).
- Keep CLI command code in `cmd/*`; reusable logic belongs in `pkg/*`.

## Testing Guidelines
- Tests use Go’s `testing` package and live in `*_test.go` files.
- Prefer table-driven tests for parser/option/state logic.
- Current examples: `pkg/libarchive/bsdtar_test.go`, `pkg/librevm/librevm_test.go`.
- Run `go test ./...` before opening a PR; add tests for touched behavior when practical.

## Commit & Pull Request Guidelines
Recent history favors short, lowercase commit subjects (for example `fix mount`, `refact`). Prefer concise imperative messages.

PRs should include:
- Clear scope and rationale.
- Linked issue(s) when applicable.
- Notes for platform-specific impact (macOS/Linux).
- Updated docs in `docs/` or `README.md` when behavior changes.
- Passing CI (`build.yml`) before merge.
